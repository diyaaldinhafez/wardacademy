# Ward Academy — دليل موجِّهات Claude Code (Session Playbook)

هذا الدليل يحوّل "موجز البناء" إلى **موجِّهاتٍ جاهزةٍ للنسخ**، جلسةً بعد جلسة. اعمل بالترتيب، ولا تنتقل لجلسةٍ قبل أن تنجح معايير قبول سابقتها.

---

## كيف تعمل (اقرأه مرّةً واحدة)

1. **ضع ملف الموجز في المشروع:** ضع `Ward_Academy_v1_Build_Brief.md` داخل مجلّد المشروع. كل موجِّهٍ أدناه يطلب من Claude Code أن يعامله **مرجعاً ملزماً**.
2. **إيقاع كل جلسة ثابت:** (أ) تلصق الموجِّه؛ (ب) يكتب لك Claude خطّةً نثراً بالعربية — **تراجعها وتوافق قبل أن يكتب أيّ كود**؛ (ج) يبني؛ (د) يكتب الاختبارات ويهاجم عمله؛ (هـ) تمرّ على معايير القبول بنفسك.
3. **لا تتجاوز جلسةً** قبل أن تتحقّق معايير قبولها.
4. **القاعدة الذهبية:** إن اقترح Claude ميزةً خارج نطاق الجلسة، أوقِفه: "خارج النطاق الآن."

## جلسة ٠ — تجهيزٌ تفعله أنت بيدك (Dashboards) قبل أيّ كود

هذه خطواتٌ خارج Claude Code، تحميك من أكبر المخاطر:

1. **سقف التكلفة (الأهمّ):** في حساب Anthropic، اضبط **حدّ إنفاقٍ شهريّ + تنبيهاً**. هذا حزام أمانك ضدّ أيّ فاتورةٍ مفاجئة، ولا يستطيع أيّ خطأٍ برمجيٍّ كسره.
2. **قاعدة البيانات:** أنشئ مشروع **Supabase في إقليم أوروبا (Frankfurt / eu-central)**. لا تختر إقليماً غير أوروبي.
3. **الرصد (اختياري الآن، مطلوب قريباً):** أنشئ حساب **Sentry**.

احتفظ بالمفاتيح (API keys) لتعطيها Claude Code ليضعها في **متغيّرات بيئةٍ على الخادم فقط**.

## جلسة ١ — الأساس + الشريحة العمودية للفرضية الأخطر

> انسخ هذا:

```
اقرأ ملف Ward_Academy_v1_Build_Brief.md في المشروع وتعامل معه كمرجعٍ ملزم لا تخرج عنه.

في هذه الجلسة نبني الأساس فقط + شريحةً عموديةً واحدة. لا تبنِ أيّ ميزةٍ أخرى.

المطلوب:
1) إعداد المشروع: Next.js + React + TypeScript + Tailwind، Mobile-First، اتجاه LTR، مع Git من البداية.
2) الهوية عبر مزوّدٍ مُدار (Supabase Auth) — لا تكتب نظام هوية يدوياً.
3) قاعدة Supabase (إقليم أوروبا) مع RLS، ونموذج بيانات أوّليّ بالطبقات: Tenant، Instructor (المعلّم كيانٌ ينتمي إليه طلابه)، Guardian (حامل الحساب المُصادَق)، Learner (مع is_minor، ملفٌّ مرتبطٌ بوليّ أمرٍ بدخولٍ بسيطٍ مُهيَّأ، لا تسجيلٌ ذاتيّ)، Objective، Item (يحمل وسم الهدف + حقل مصدر/إسناد اختياري فارغ الآن). نموذج الهوية مرتكزٌ على وليّ الأمر: نظام هويةٍ واحد (Managed Auth)، وعلاقة وليّ أمرٍ واحد ← عدّة طلاب، بموافقةٍ مخزَّنةٍ بختمٍ زمنيٍّ لكلّ ابنٍ على حدة. لا حساب مشترك، ولا وصول متكرّر بروابط بلا مصادقة (الروابط المُوقَّعة للأفعال لمرّةٍ واحدةٍ قبل الحساب فقط). اترك مكاناً لأبٍ اختياريٍّ Organization مستقبلاً دون بناء أيّ ميزةٍ مؤسسية.
4) عزل المستأجر صارم: tenant_id على كل جدول، تمرير سياق المستأجر عبر JWT، ولفّ استدعاءات الهوية في سياسات RLS كـ (select auth.uid()).
5) مفتاح Claude API في الخادم حصراً (متغيّر بيئة)، ويُمنع أيّ نداءٍ من الواجهة.
6) التوليد خلف حدّ خدمةٍ واحدٍ نظيف (generation service) لا منثوراً في الكود.
7) الشريحة العمودية: شاشةٌ بسيطةٌ يدخل فيها المعلّم هدفاً تعليمياً واحداً، فتُنادي خدمةَ التوليد على الخادم، فيولّد Claude سؤالاً تدريبياً واحداً أصلياً (لا استنساخ من أيّ كتاب)، ثم يراه المعلّم ويعتمده. لا يصل أيّ شيءٍ لأيّ طالبٍ قبل اعتماد المعلّم.

أولاً: اكتب لي الخطّة نثراً بالعربية (نموذج البيانات، سياسات العزل، كيف يعمل التوليد) وانتظر موافقتي قبل كتابة أيّ كود.
```

**بعد البناء، قل له:**
```
اكتب اختباراً آلياً يتأكّد أن مستأجراً لا يصل بيانات مستأجرٍ آخر، وصِف لي بالعربية ماذا يفحص. ثم هاجِم عملك: هل يمكن أن يتسرّب مفتاح Claude للواجهة؟ هل يصل مخرجٌ للطالب دون اعتماد؟
```

**معايير قبول جلسة ١ (تفحصها بنفسك):**
- [ ] تستطيع تسجيل دخول معلّمٍ، وإدخال هدف، وتوليد سؤالٍ واحدٍ أصليّ، واعتماده.
- [ ] جودة السؤال المولَّد تُرضيك فعلاً. **(إن لم تُرضِك — توقّف هنا وعالِجها قبل أيّ شيء؛ هذه فرضيتك الأخطر.)**
- [ ] مفتاح Claude لا يظهر في المتصفّح.
- [ ] الطالب ينتمي لمعلّمٍ لا "للتطبيق".

## جلسة ٢ — سقالات الأمان (قبل الميزات الجميلة)

> انسخ هذا:

```
مرجعك Ward_Academy_v1_Build_Brief.md (البند ٧). في هذه الجلسة نبني سقالات الأمان فقط، لا ميزات.

المطلوب:
1) اختبارات عزلٍ آلية (Isolation Tests) تُشغَّل تلقائياً عند كل تغيير: لا يرى مستخدمٌ بيانات غيره، ولا يفتح رابطٌ لطالبٍ لطالبٍ آخر. صِف كل اختبارٍ بالعربية.
2) ربط Sentry لرصد الأخطاء وإظهار الفشل الصامت.
3) فصلُ بيئة تجربة (staging) عن بيئة الإنتاج، بحيث لا تلمس التجربة أيّ بيانات حقيقية.
4) عدّاد استهلاك Claude لكل مستخدم في جدولٍ أراه، مع سقفٍ برمجيٍّ يكمّل سقف الحساب.
5) أداةٌ لتوليد روابط مُوقَّعة (signed)، محدودة الصلاحية، ومربوطةٍ بطالبٍ واحد — سنستخدمها لاحقاً.

اكتب الخطّة نثراً بالعربية أولاً وانتظر موافقتي.
```

**معايير القبول:**
- [ ] اختبارات العزل تعمل وتنجح، وتفشل عمداً لو كُسِر العزل.
- [ ] خطأٌ مُفتعَلٌ يظهر في Sentry.
- [ ] التجربة في staging لا تمسّ بيانات حقيقية.

## جلسة ٣ — حلقة العمل (الواجبات والتصحيح)

> انسخ هذا:

```
مرجعك الموجز (البند ٣.٣ و٣.٤). نبني حلقة العمل فقط.

المطلوب:
1) المساران: طالب تقويةٍ مدرسية (أهدافٌ مرنةٌ يدخلها المعلّم لكل طالب) + طالب CEFR (سُلّم مستوياتٍ ثابت). نموذج الأهداف يَمرن للاثنين.
2) إنشاء الواجب بطريقتين: (أ) مسار الذكاء: المعلّم يدخل الموضوع ويحدّد أنواع العناصر المطلوبة قبل التوليد (اختيار من متعدّد، فراغات، صح/خطأ، توصيل، إجابة مفتوحة، تسجيل صوتيّ من الطالب…) ← توليد محتوىً أصليّ بهذه الأنواع ← مراجعة واعتماد ← إرسال. (ب) مسار يدويّ بلا ذكاء: نموذجٌ بحقول (وحدة/درس/صفحة/تمرين) + رفع صورة/ملف من داخل المنصّة كمرفقٍ خاملٍ معزولٍ تماماً (تخزينٌ وعرضٌ فقط، خارج محرّك التوليد).
3) أنواع العناصر والتصحيح: موضوعية (تصحيح آلي)؛ الذكاء يقيّم النصّ المكتوب؛ كتابة مفتوحة (الذكاء يولّد مسوّدة تقييم ← المعلّم يعتمد ← الطالب يرى)؛ وتسجيل صوتيّ يأخذ الطالب علامته بمجرّد الإرسال (علامة إتمام) ويستمع إليه المعلّم للملاحظة، بلا تقييمٍ آليٍّ أو درجةٍ على المحتوى الصوتيّ.
4) سطح الطالب: حلٌّ داخل المنصّة، أو رفع صورة حلّه، أو تسجيل صوت.
5) كل سؤالٍ موسومٌ بهدفه التعليمي منذ إنشائه.

اكتب الخطّة نثراً بالعربية أولاً وانتظر موافقتي. بعد البناء، هاجِم عملك: هل تدخل مرفقات المعلّم محرّك التوليد؟ هل يصل تقييمٌ ذاتيٌّ للطالب دون اعتماد؟
```

**معايير القبول:**
- [ ] كلا مساري إنشاء الواجب يعملان.
- [ ] صور/ملفات المعلّم لا تدخل التوليد إطلاقاً.
- [ ] تقييم الكتابة المفتوحة لا يصل الطالب قبل اعتماد المعلّم.
- [ ] كل سؤالٍ مربوطٌ بهدف.

## جلسة ٤ — التقدّم وتقرير الجلسة

> انسخ هذا:

```
مرجعك الموجز (البند ٣.٤ و/و). نبني التتبّع والتقارير فقط.

المطلوب:
1) تتبّع التقدّم: الحضور، درجات الواجبات والاختبارات، نقاط القوة/الضعف محسوبةً من صواب/خطأ العناصر مجمَّعةً حسب الهدف، وملخّصٌ سرديٌّ يولّده الذكاء من التاريخ المتراكم (لا نموذجٌ دائمٌ في الخلفية — تلخيصٌ عند الطلب). ولطلاب CEFR: تقدّمٌ حسب المستوى.
2) اعرض الأرقام بصدق ("6 من 8") لا نِسَبَ إتقانٍ موهِمة.
3) تقرير الجلسة: نموذجٌ منظّمٌ سريعٌ (اختياراتٌ متعدّدة + خانة نصٍّ اختيارية) ← زرّ توليد ← الذكاء يصوغ التقرير ← المعلّم يعتمد ← يُشارَك مع وليّ الأمر. خزّن اختيارات النموذج نفسها كبياناتٍ تُغذّي التقدّم، لا التقرير السرديّ فقط.

اكتب الخطّة نثراً بالعربية أولاً وانتظر موافقتي.
```

**معايير القبول:**
- [ ] القوة/الضعف تعكس الأهداف الموسومة فعلاً.
- [ ] تقرير الجلسة لا يصل وليّ الأمر قبل اعتماد المعلّم.
- [ ] اختيارات النموذج محفوظةٌ كبيانات.

## جلسة ٥ — البوّابة الأمامية (التسجيل، الحجز، اختبار التحديد)

> انسخ هذا:

```
مرجعك الموجز (البند ٣.١ و٣.٢، والوصلات ٣ و٧ و٨). نبني البوّابة الأمامية فقط.

المطلوب:
1) صفحة هبوطٍ واحدة + دعم ربط دومين مخصّص.
2) نموذج تسجيلٍ يملؤه وليّ الأمر، يلتقط موافقته الأبويّة صراحةً ويخزّنها مع ختمٍ زمنيّ قبل تفعيل حساب القاصر. اجمع أقلّ بياناتٍ ممكنة.
3) حجز جلسةٍ تجريبية: المعلّم يحدّد مواعيد توفّره (متكرّرة أسبوعياً أو مفردة)، ويختار وليّ الأمر منها. خزّن كل التوقيتات بـ UTC واعرضها بالتوقيت المحلّي لكل مستخدم، وامنع الحجز المتعارض.
4) عند التسجيل تظهر بيانات الطالب للمعلّم مباشرةً (سواءٌ حُجِز موعدٌ أم لا)؛ وإن حُجِز ظهرت حالة الحجز. زرّ "توليد اختبار تحديد مستوى": الذكاء يولّده من بيانات الطالب ← المعلّم يعتمده ← يُشارَك رابطٌ مُوقَّعٌ محدود الصلاحية مربوطٌ بهذا الطالب فقط (واتساب) ← الطالب يحلّه ← النتيجة والتقييم للمعلّم ← يعتمد ← يشارك مع وليّ الأمر. يعيد استخدام محرّك التوليد/التقييم نفسه.
5) تقرير الجلسة التعريفية (بنفس مبدأ تقرير الجلسة) + توليد الخطة الدراسية: لمسار CEFR من بيانات التحديد + تقرير الجلسة؛ وللمسار المدرسيّ يرفع المعلّم فهرس الكتاب (ملف/صورة) فيولّد الذكاء خطةً أصلية منه ثم يعتمدها المعلّم. الفهرس يُستخدَم مرّةً لاستخراج البنية/الأهداف ولا يُخزَّن كمصدر إسنادٍ دائم.

اكتب الخطّة نثراً بالعربية أولاً وانتظر موافقتي. بعد البناء، هاجِم عملك: هل يفتح رابط اختبارٍ لطالبٍ آخر؟ هل تظهر المواعيد بتوقيتٍ خاطئ؟ هل يُفعَّل حساب قاصرٍ بلا موافقة؟
```

**معايير القبول:**
- [ ] لا يُفعَّل حساب قاصرٍ قبل تخزين موافقة وليّ الأمر.
- [ ] المواعيد تظهر بالتوقيت المحلّي، ولا تعارض في الحجز.
- [ ] رابط اختبار التحديد لا يعمل لطالبٍ آخر وتنتهي صلاحيته.

## جلسة ٦ — لوحة وليّ الأمر (قراءة فقط)

> انسخ هذا:

```
مرجعك الموجز (البند ٣.٥). ابنِ لوحةً لوليّ الأمر للقراءة فقط: تقدّم أبنائه ودرجاتهم وحالة واجباتهم، **مع التنقّل بين أبنائه المتعدّدين** إن وُجدوا، والتقارير التي شاركها المعلّم. لا تواصل داخل المنصّة (يبقى عبر واتساب). تأكّد أن وليّ الأمر يرى أبناءه هو فقط لا أبناء غيره.

اكتب الخطّة نثراً بالعربية أولاً وانتظر موافقتي.
```

**معايير القبول:**
- [ ] وليّ الأمر يرى بيانات ابنه فقط.
- [ ] اللوحة للقراءة فقط (لا تعديل).

## قبل أول طالبٍ حقيقي (لا تتجاوزه)

1. مُرّ على **كل معايير القبول** في موجز البناء (البند ١٠) بنداً بنداً بنفسك.
2. احصل على **مراجعةٍ أمنيةٍ بشريةٍ واحدة** موجَّهةٍ لأربعة: المفتاح في الخادم، العزل صحيح، الهوية عبر المزوّد المُدار، الروابط المشتركة مُوقَّعةٌ محدودة. اعتبرها جزءاً من تكلفة الإطلاق لا بنداً تؤجّله.
3. ابدأ ببيانات تجربةٍ وهمية، ثم بطالبٍ واحدٍ حقيقيٍّ قبل التوسّع.

## مبادئ تحملها في كل جلسة

- **الخطّة نثراً أولاً، ثم البناء.** أنت تُقيّم التفكير لا الكود.
- **جلسةٌ واحدةٌ = نطاقٌ واحد.** أوقِف أيّ توسّعٍ خارج الجلسة.
- **اطلب من Claude أن يهاجم عمله** بعد كل جزء.
- **لا تتقدّم قبل نجاح معايير القبول.**
- إن تعثّرت جلسةٌ وكبرت، اطلب من Claude تقسيمها إلى خطواتٍ أصغر — لا تكدّس.
