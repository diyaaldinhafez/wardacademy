## 1) What the project is
Ward Academy — an English-learning platform for children **9–13**, one-on-one live sessions with a teacher, assisted by AI **with the teacher approving every output before it reaches the child**. One curriculum: **"Ward Curriculum"** — a platform-specific curriculum measured against **CEFR (A1→B1)** — 30 units · 215 objectives, complete and frozen in `Ward_Curriculum_Master_Reference.md`. (The "school curriculum" track was retired permanently on 2026-06-20.)

## 6) How we work (the sync ritual)
- **One source of truth per domain:** curriculum → Curriculum Reference · constitution → Foundation · design → Design Brief · pulse/status → this file. Truth is not duplicated across two files.
- **Start of any session** (Code/Design): "read `START_HERE.md` first before you begin".
- **End of any Code session**: "update `START_HERE.md`: status + session log + any new decision".
- **A decision is not adopted until it is written in its domain's source of truth**, then referenced here. The direction: decision → domain document → pulse here (not the reverse silently).
- **When returning to Claude (Chat)**: share the latest version of this file.
- **(Methodological lesson — a migration ships with its code):** `db:migrate` hits the production database immediately; so any gate carrying a migration is **pushed and deployed in the same round** — no local unpushed commits left behind, otherwise a "database ahead of deployed code" window opens that breaks production. (Happened 2026-06-25 with 0062/0063.)
- **(Methodological lesson — every deploy ends with a live human check):** "green build" proves compilation, not live behavior; **the final verification is by eye on the deployed site**, not by a report alone.
- **(Methodological lesson — "build green" hides whole classes of faults · confirmed 2026-06-27):** translation faults (raw i18n keys), font faults, and display/layout issues **do not break the build** — they surface only at runtime. Every affected surface needs a **live visual check** + confirming **Vercel Ready = HEAD**, not just a green compile.
- **(Methodological lesson — font bugs need both-direction visual checks):** verify *both* "Arabic not broken" *and* "English now correct". Rule-level / CSS-resolution proof can pass while the visual still fails — the `Nunito Fallback` (local Arial) Arabic interception was invisible to rule-level analysis and only showed by eye (2026-06-27).
- **(Methodological lesson — delete a file together with its consumer in the SAME commit):** a failed `git add` on a deleted-file pathspec silently splits the change and ships a broken build (missing-module). Use `git add -A` (or `git rm`) and check `git status` before committing.
- **(Methodological lesson — read deploy health from the TOP of the Deployments list):** the newest deployment is live; judging health from a stale failed deployment opened by name gives a false negative (2026-06-27).

> - **Layer 1 (objective):** the **65/35 decaying average over the objective's repeated assessments is CORRECT and intentional** — an objective is assessed **multiple times** across lessons (teacher + auto-graded homework + unit test), so the decay absorbs fluctuation and allows full recovery. **Do NOT change to last-write/replace.** *(This overturns the earlier instinct to "replace at L1"; the per-objective diagnostic's "accumulation" is the model working — the real fault is the display, see below.)*
> - **Layer 2 (unit / "unit flower"):** simple average of **ALL** the unit's objective values (skills mixed) — per reference §5. (Currently implemented as such; no per-unit, per-skill value exists, which is fine for the unit flower.)
> - **Layer 3 (skill petal, across started units):** reference §1/§5 says "mean of that skill's objectives across started units" but does **NOT** specify simple-vs-decaying. **OPEN QUESTION (resolve at design time):** should the cross-unit skill petal be a **simple mean** (current code) or a **decaying average across units**?
