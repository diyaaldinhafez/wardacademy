## 1) What the project is
Ward Academy — an English-learning platform for children **9–13**, one-on-one live sessions with a teacher, assisted by AI **with the teacher approving every output before it reaches the child**. One curriculum: **"Ward Curriculum"** — a platform-specific curriculum measured against **CEFR (A1→B1)** — 30 units · 215 objectives, complete and frozen in `Ward_Curriculum_Master_Reference.md`. (The "school curriculum" track was retired permanently on 2026-06-20.)

## 3) Pinned decisions (not reopened without cause)
- Identity is **guardian-anchored** (one account per guardian, multiple children, consent required per account).
- **The teacher approves every AI output** before it reaches the child (the core trust model).
- **Progress (the new model):** objective-level assessment with four states (seed/bud/balloon/bloom) decided by the teacher · a **65/35 decaying average** at the objective level · a simple automatic roll-up for the unit (**the unit flower**) and the major flower (skills across started units). Scope A1→B1 on "Ward Curriculum". **Source of truth: `Ward_Curriculum_Master_Reference.md`.** *(The new model is fully adopted — teacher + auto via the catalog; the old model is retired. The migration completed 2026-06-24.)*
- **Europe data + RLS isolation** between students.
- **Original generation** of content (no ingestion of copyrighted books).
- **Ages 9–13**, **English-first** across the platform: the internal surfaces (child/teacher/admin) are English; the parent surfaces are bilingual with an "ع" toggle; and communication is in the parent's saved language (see the language policy below).
- **In the parent-facing UI:** the AI is "a support for the teacher and student", no more; its technical details stay in the teacher UI. Western numerals, no emoji, first names only.
- **Strict separation of responsibilities:** Admin (`/admin`) owns the pre-learning funnel + billing + cases + teacher oversight; the teacher (`/studio`) is for teaching and setting her own availability only; the roles do not overlap.
- **The teacher owns her availability:** she sets/edits it from her account, and Admin views her schedule read-only.
- **Mobile-first across the entire build journey:** every screen is designed and verified on mobile first (the shells use a bottom bar on phones).
- **Language policy (English-first · i18n · decision 2026-06-24):** all surfaces are English/LTR by default (the child surface is pure, no toggle); Arabic is an optional layer for the parent surfaces (landing · enrollment · guardian dashboard · reports/communication) via a toggle + one-click translation for reports. **i18n from day one** (`en.json` first · `ar.json` drop-in). (Details: `Foundation §13`.) **✅ i18n fully complete (2026-06-25):** all surfaces (1–7) converted to `next-intl`, the old system retired and deleted, one system (next-intl + the `LOCALE` cookie) — see "Technical facts §Language" and the banner below.
- **Semantic colors, not purple only:** purple is the hero, but use green (success/completion), apricot (a warm invitation / next step), and rose (gentle errors/deletion, limited, no harsh red).
- **Design workflow (pinned decision):** Claude Design is for reusable visual identity only (logo · colors · font · icons · the flower/spark) — **no screens**. Screens and pages are built in Claude Code directly. **The deployed code is the source of truth for screens and identity once applied;** code edits are not fed back to Design (Design is a generation tool, not an archive). The new flower (four petals + a seed) and the new logo from Design = the adopted identity that the code must catch up to.
> Full details in: `Ward_Academy_Foundation.md` and `Ward_Academy_Design_System_Brief.md`.

- **End of any Code session**: "update `START_HERE.md`: status + session log + any new decision".
- **A decision is not adopted until it is written in its domain's source of truth**, then referenced here. The direction: decision → domain document → pulse here (not the reverse silently).
- **When returning to Claude (Chat)**: share the latest version of this file.
- **(Methodological lesson — a migration ships with its code):** `db:migrate` hits the production database immediately; so any gate carrying a migration is **pushed and deployed in the same round** — no local unpushed commits left behind, otherwise a "database ahead of deployed code" window opens that breaks production. (Happened 2026-06-25 with 0062/0063.)
- **(Methodological lesson — every deploy ends with a live human check):** "green build" proves compilation, not live behavior; **the final verification is by eye on the deployed site**, not by a report alone.

## 8) Pinned technical facts
- **Framework:** Next.js 16 (App Router) + React 19 + TypeScript + Tailwind v4 + Turbopack.
- **Node 20 mandatory** (Node 24 is incompatible with the edge-runtime in Next 16) — pinned in `.nvmrc`.
- **Deployment:** Vercel on `ward.academy`. **Repo:** github.com/diyaaldinhafez/wardacademy, branch `main`.
- **Deployment status (updated on every code push):** the live build on Vercel = code `f7035b2` (VocabCounter removal — the last app change; the documentation commits after it don't change the build) · `HEAD` in sync with `origin/main` · production migrations **0055→0063** (no gap, `objectives` dropped) · **no mismatch window between the production database and the deployed code.**
- **Language (i18n — complete):** `next-intl` v4 only — the language is in the `LOCALE` cookie (default `en`, no path prefixes) via `i18n/request.ts`. The internal surfaces (child/teacher/admin/video) are **forced English** (provider `locale="en"` + `getTranslations({locale:"en"})`); the parent surfaces (landing/enrollment/test/booking/guardian/set-password) are **bilingual, cookie-driven** + an "ع" toggle with dynamic `dir`/`lang`. `messages/{en,ar}.json` (key parity checked programmatically; the forced surfaces are en-only by design). **The old system (`LanguageProvider`/`useT`/`LangToggle`/`lib/i18n`/`lib/content`) is retired and fully deleted.** **Communication** reaches the parent in their saved language `leads.guardian_locale` via `getTranslations({locale: parentLocale})` (overriding the teacher/admin UI's forced locale), with `ar` fallback.
- **Database:** Supabase (PostgreSQL, EU/Frankfurt) + RLS. **63 migrations** (`supabase/migrations/*.sql`) applied via `npm run db:migrate` (a pg script, the Session pooler string in `.env.local` as `SUPABASE_DB_URL`). The latest are **0060–0063** (plan unification on the catalog: confirmed level 0060 · `items.objective_id`→catalog 0061 · `submissions.objective_id` nullable+trigger decoupled 0062 · drop the `objectives` table 0063). Before them 0055–0059 (catalog/assessment model/old-model retirement/parent language).
- **Role guards (server):** `lib/auth.ts` → `assertAdmin()` / `assertInstructor()` at the head of every action that uses the service role (RLS alone is not enough since the service key bypasses it). The service-role client is in `lib/supabase/admin.ts`.
- **Email:** Resend (`lib/email.ts`, the `ward.academy` domain DKIM/SPF-verified) — booking confirmation + invite link + intro-session report; it degrades harmlessly if the key is absent. **Dates/time zones:** `luxon` + `tenants.timezone`.
- **Supabase clients:** `lib/supabase/{client,server,admin}.ts` + `proxy.ts`/`lib/supabase/session.ts` (session refresh and protection of `/studio` `/learn` `/guardian`).
- **AI:** `lib/generation/service.ts` (server-only) via `@anthropic-ai/sdk` (model `claude-sonnet-4-6`): items/questions, a session report from the teacher's inputs, a unit test from the objectives, internal diagnostics. The Anthropic spend cap is set ($5 trial).
- **Verification:** `npm run build` before any push. `.env.local` is outside Git (Supabase + Anthropic + DB URL).
